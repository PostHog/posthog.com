When you use PostHog on both the frontend (JavaScript SDK) and backend (Python, Node, etc.), getting user identity right requires some coordination. This guide covers the most common patterns and pitfalls.

## The basics

The PostHog JavaScript SDK automatically generates an anonymous `distinct_id` for each visitor. When a user logs in, you call `posthog.identify('your-user-id')` to link that anonymous ID to their real identity.

On the backend, you pass the user's ID directly to `capture()`:

```python
posthog.capture('your-user-id', 'subscription_created')
```

As long as both sides use the **same** `distinct_id`, events land on the same person. Simple enough — but it gets complicated with multi-step authentication.

## The multi-ID problem

Many apps use an authentication provider (Firebase, Auth0, Supabase) that assigns its own user ID, separate from your internal database ID. This creates three IDs for a single user:

1. **Anonymous ID** — auto-generated by posthog-js (e.g., `019cf913-a889-...`)
2. **Auth provider ID** — from Firebase/Auth0 (e.g., `K4inej7WcVeT...`)
3. **Internal user ID** — from your database (e.g., `user_123`)

Here's a typical signup flow that causes problems:

```
Frontend: User lands on /signup           → anonymous ID created
Frontend: User verifies email             → posthog.identify('firebase-uid')     ✅ links anon → firebase
Server:   Post-signup hook fires          → posthog.capture('user_123', 'signup') ❌ creates NEW person
Frontend: App loads with internal user ID → posthog.identify('user_123')          ❌ merge BLOCKED
```

The merge from `firebase-uid` → `user_123` is blocked because the person associated with `firebase-uid` is already "identified" from the earlier `identify` call. PostHog won't merge two identified persons via `identify` — this is a safety feature to prevent accidentally combining different users.

**Result:** Two separate persons. The pre-signup journey is on one, post-signup activity is on another.

## Fix 1: Server-side alias before events

The most reliable fix is to call `alias` on your server to link the new internal ID to the previous ID **before** sending any events:

```python
# In your signup handler, BEFORE any capture() calls:
posthog.alias(
    previous_id='firebase-uid',
    distinct_id='user_123',
)

# Now events with user_123 land on the correct person
posthog.capture('user_123', 'signup_completed')
```

```node
// Node.js equivalent
client.alias({
    distinctId: 'user_123',
    alias: 'firebase-uid',
})

client.capture({
    distinctId: 'user_123',
    event: 'signup_completed',
})
```

For this to work, your server needs access to the previous distinct ID (the Firebase UID or the anonymous ID). Pass it through your signup flow — for example, include it in the signup request from the frontend.

## Fix 2: Tracing headers for API-triggered events

For server events that are triggered by frontend API calls (user clicks a button → your API handles the request → server captures an event), you can use tracing headers:

```js-web
posthog.init('<ph_project_api_key>', {
    api_host: '<ph_client_api_host>',
    __add_tracing_headers: ['your-api-domain.com'],
})
```

This automatically injects `X-POSTHOG-SESSION-ID` and `X-POSTHOG-DISTINCT-ID` headers on requests to your backend. The PostHog Django middleware picks these up and uses the frontend's identity for server-side events.

This works for events like `api_key_created` where the trigger is a user action, but **not** for webhook-triggered events (like a Firebase post-creation hook) since those requests don't come from the browser.

## Fix 3: Retroactive merge with $merge_dangerously

If you've already hit the split-person problem, you can retroactively merge them:

```python
posthog.capture(
    distinct_id='user_123',
    event='$merge_dangerously',
    properties={'alias': 'firebase-uid'},
)
```

> **Warning:** `$merge_dangerously` is irreversible and bypasses all safety checks. Use it for one-off recovery, not as a regular pattern.

## Summary

| Approach | Best for | Limitation |
|---|---|---|
| Server-side `alias` | Signup flows with multiple IDs | Requires passing previous ID to server |
| Tracing headers | Server events from frontend API calls | Doesn't work for webhooks/background jobs |
| `$merge_dangerously` | Retroactive fixes | Irreversible, no safety checks |

The golden rule: **link all your IDs together before sending events with a new ID.** If your server creates a new user ID, call `alias` to connect it to the previous ID before capturing any events with it.
