Spans are units of work within an LLM [trace](/docs/llm-analytics/traces). These are events that represent individual operations and discrete durations within your LLM application, like function calls, vector searches, or data retrieval steps, providing granular visibility into the execution flow.

PostHog captures spans to track atomic operations that make up your LLM workflow. For example:

- **[Generations](/docs/llm-analytics/generations)** - LLM calls and interactions
- **Vector database searches** - Document and embedding retrieval
- **Tool/function calls** - API calls, calculations, database queries  
- **RAG pipeline steps** - Retrieval, reranking, context building
- **Data processing** - Validation, chunking, formatting

For technical implementation details, see [manual capture](/docs/llm-analytics/installation/manual-capture).

## Filtering spans

Use the filter controls to find specific operations in the spans list. You can filter by:

### Span name

Filter spans by name using string matching:

- **Equals / Not equals** – Match an exact name
- **Contains / Not contains** – Match names that include a substring
- **Matches regex / Not matches regex** – Match using regular expression patterns

### Span kind

Filter by OpenTelemetry span kind, which indicates the span's role in a distributed trace:

| Kind | Description |
|------|-------------|
| Unspecified | Default when no kind is set |
| Internal | Internal operation within an application |
| Server | Handles an incoming request (e.g., HTTP server) |
| Client | Makes an outgoing request (e.g., HTTP client, database call) |
| Producer | Creates a message for async processing |
| Consumer | Processes a message from a queue |

### Status code

Filter spans by completion status:

- **OK** – The operation completed successfully
- **Error** – The operation failed
