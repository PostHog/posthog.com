Definitions lets you manage PostHog dashboards, insights, feature flags, and endpoints as TypeScript files in your repo. Commit them alongside your application code and run `npx posthog-definitions apply` to sync them to your project.

## Why use this

Most PostHog setups are clicked together in the UI. That works until you need to:

- **Review changes before they ship.** A dashboard or flag change goes through the same PR review as the code that depends on it, instead of silently mutating in production.
- **Keep environments in sync.** Defining a dashboard once and applying it to staging and prod (and a teammate's dev project) is one `apply` call away – no manual rebuilding per project.
- **Roll back when something breaks.** If a "small tweak" to a metric breaks an exec dashboard, `git revert` + `apply` restores it. The UI has no undo.
- **Treat analytics as code.** The events your app emits live in your repo. The dashboards and flags that depend on those events should too – so a rename or removal shows up as a single, reviewable diff.

If none of these are pain points yet, the UI is fine. Reach for Definitions when ad-hoc changes start having consequences.

</CalloutBox>

## Install

```bash
npm install --save-dev @posthog/definitions
```

The package ships a `posthog-definitions` CLI that runs via `npx`.

## Authenticate

```bash
npx posthog-definitions login
```

This opens a browser, completes an OAuth flow, and prompts you to pick a project. Credentials are saved to `~/.config/posthog-definitions/config.json`.

To switch projects without re-authenticating, run `npx posthog-definitions login --switch-project`. To clear credentials, run `npx posthog-definitions logout`.

For CI, set `POSTHOG_PERSONAL_API_KEY` and `POSTHOG_PROJECT_ID` in the environment instead. They take precedence over the stored OAuth token.

<CalloutBox icon="IconInfo" title="API key scopes" type="fyi">

Personal API keys used by `apply` need the `dashboard:write` and `insight:write` scopes. Create or update keys at [Personal API Keys](https://us.posthog.com/settings/user-api-keys).

</CalloutBox>

## Define a dashboard

Create `posthog/dashboards/growth.ts`:

```ts
import { dashboard, insight, trends, text } from "@posthog/definitions"

const signups = insight({
  key: "weekly-signups",
  name: "Weekly signups",
  query: trends({
    series: [{ event: "user signed up", math: "total" }],
    interval: "week",
    dateRange: { date_from: "-90d" },
  }),
})

export default dashboard({
  key: "growth",
  name: "Growth",
  description: "Top-of-funnel and activation",
  pinned: true,
  tags: ["growth"],
  tiles: [
    { insight: signups, layout: { x: 0, y: 0, w: 6, h: 4 } },
    text({ body: "Updated weekly.", layout: { x: 0, y: 4, w: 12, h: 1 } }),
  ],
})
```

Each file under `posthog/` must `export default` a single dashboard or insight. The `key` is the stable identity – the CLI uses it to match local files to server resources across runs.

## Apply

Preview what would change without making API calls:

```bash
npx posthog-definitions apply --dry-run
```

Sync to PostHog:

```bash
npx posthog-definitions apply
```

`apply` is idempotent – re-running with no changes is a no-op.

The CLI only touches resources it created (tagged `iac:dashboards:<key>` or `iac:insights:<key>`). Dashboards and insights you built in the UI are invisible to `apply`: not modified, not deleted, not warned about. You can adopt Definitions one dashboard at a time without risk to the rest of the project.

By default `apply` never deletes anything. Pass `--prune` to delete IaC-tagged resources whose source files have been removed.

## Pull

Bootstrap a definitions directory from an existing project:

```bash
npx posthog-definitions pull
```

This writes a TypeScript file per dashboard to `posthog/dashboards/`. Review with `git diff` before committing – existing files with the same name are overwritten.

## What you can define

| Resource | SDK | `apply` |
| --- | --- | --- |
| Dashboards | `dashboard()` | ✅ |
| Insights | `insight()` with `trends()` or `hogql()` | ✅ |
| Feature flags | `featureFlag()` | Coming soon |
| Endpoints | `endpoint()` | Coming soon |

`funnel()`, `retention()`, `paths()`, `stickiness()`, and `lifecycle()` query constructors are planned post-MVP.
