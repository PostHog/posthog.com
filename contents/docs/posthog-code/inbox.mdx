PostHog **Inbox** is your command center for self-driving development. It uses signals from your production data - like errors appearing, users stuck in session recording, or support tickets – to automatically research issues, write code fixes, and open pull requests.

Instead of reactive firefighting, Inbox moves your workflow to a proactive merging model where AI agents do the heavy lifting of investigation and implementation.

## Signal sources

Inbox identifies potential work by monitoring **signal sources**. These sources provide the raw data that AI agents use to research and triage issues.

### Configuration

To manage your sources:
1. Open **Inbox** in PostHog Code
2. Click the **Settings** icon (or the **Signal Sources** tab)
3. Toggle the sources you want to enable and follow the authentication prompts for external services

> **Note:** You don't need to manually configure MCP for Inbox sources. Once your repository is connected in PostHog Code, research tasks can run automatically.

### Internal PostHog sources

These sources are available by default if you're already using PostHog.

#### Error Tracking

Monitors your production exceptions. Inbox triggers reports when:
- A new unique error appears
- An error reopens after being marked resolved
- There's a significant spike in error volume for an existing issue

#### Support conversations

Analyzes user support tickets from PostHog's built-in [support features](/docs/support). It identifies actionable bug reports or feature requests within user chats.

#### Session Replay

(Beta) Uses AI to analyze [session recordings](/docs/session-replay) for anomalous user behavior, UX friction points, or rage clicks that don't trigger an error log.

### External integrations

Connect external tools to Inbox to consolidate your engineering work in one place.

#### GitHub Issues

Inbox monitors your repositories for new issues. When a GitHub issue is opened, Inbox automatically kicks off a **research task** to see if it can propose a fix.

#### Linear and Zendesk

By connecting Linear (for project management) or Zendesk (for customer support), Inbox can:
1. Read the ticket or issue context
2. Search your codebase for relevant logic
3. Propose a fix that references the original ticket ID

## Triage and prioritization

Inbox uses AI to cluster raw signals into actionable **reports** and rank them so you know what to work on first.

### Understand reports

A **report** is a high-level summary of an issue. Each report contains:
- **Problem description** - A clear description of the bug or issue
- **Priority** - Ranked from P0 (critical) to P3 (low)
- **Actionability** - Whether the AI agent has found a clear path to a code-based fix

### How Inbox determines priority

Instead of looking at error frequency alone, Inbox uses a product-aware prioritization engine that analyzes:
1. **Code importance** - Does the issue occur in a hot path like your checkout, signup, or billing logic?
2. **User impact** - How many unique users are affected? Are they on a high-value billing plan?
3. **Severity** - Is the product completely broken for these users, or is it a minor UX issue?

### Triage workflow

1. **P0 and P1 (immediate action)** - These should be your first priority. If they're marked as **Actionable**, an agent has likely already opened a pull request.
2. **Actionable reports** - Focus on these even if the priority is lower. The agent has already done the research and implementation work for you.
3. **Needs research** - If a report isn't yet actionable, trigger a **research task** to have an agent investigate the codebase.

### Assign owners

Inbox automatically suggests reviewers and owners for each report based on:
- **Git blame** - Who recently touched the affected lines of code
- **Historical ownership** - Who has resolved similar issues in this part of the codebase before

## Research tasks

When a signal (like an [Error Tracking](/docs/error-tracking) exception or a GitHub issue) arrives, an agent investigates automatically.

### Codebase context (MCP)

The agent uses a [Model Context Protocol (MCP) server](/docs/ai-engineering/mcp) to interact with your codebase. This enables it to:
- Search for specific symbols, classes, or functions
- Read file contents
- List directory structures

### Root cause analysis

The agent combines the production signal (e.g., a stack trace from Error Tracking) with codebase access to identify the root cause.

- **Example** - An agent sees a `DoesNotExist` error in production data and finds the exact line in `billing_manager.py` where a database query is missing a null check.

## Implementation tasks

Once research is complete and a fix is identified, the agent moves to implementation.

### Sandboxed execution

The agent clones your repository into a secure, sandboxed environment. It then:
1. Creates a new git branch (e.g., `ai-fix/billing-null-check`)
2. Applies the code changes
3. Runs any local tests (if configured)

### Pull request generation

The agent pushes the branch to your remote (e.g., GitHub) and opens a pull request.
- **Description** - The PR includes an AI-generated summary of the problem, the proposed change, and the research reasoning
- **Labels** - PRs are automatically labeled with `LLM context` or `PostHog Agent` to notify reviewers

## Audit the agent

Every action the agent takes is logged. View the **Agent Log** in Inbox to see:
- Which files the agent read
- Which search queries it ran
- How it arrived at the final implementation logic

This transparency ensures you can trust the process even as it automates the tedious parts of your workflow.
