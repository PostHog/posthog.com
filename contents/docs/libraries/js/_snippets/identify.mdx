Using `identify`, you can capture identified events associated with specific users. This enables you to understand how they're using your product across different sessions, devices, and platforms.

```js-web
posthog.identify(
    'distinct_id', // Required. Replace 'distinct_id' with your user's unique identifier
    { email: 'max@hedgehogmail.com', name: 'Max Hedgehog' },  // $set, optional
    { first_visited_url: '/blog' } // $set_once, optional
);
```

Calling `identify` creates a person profile if one doesn't exist already. This means all events for that distinct ID count as identified events.

You can get the distinct ID of the current user by calling `posthog.get_distinct_id()`.

### Controlling feature flag reload behavior

By default, calling `identify` with a new distinct ID reloads feature flags automatically. This reload happens before the person merge from the `$identify` event propagates server-side, which can cause users to be re-bucketed into different flag variants.

To prevent this, pass `disableFeatureFlagReload: true` as an option:

```js-web
posthog.identify(
    'authenticated-user-id',
    { email: 'max@hedgehogmail.com' },
    undefined, // $set_once properties
    { disableFeatureFlagReload: true }
);
```

When disabled, feature flags refresh on the next session or when you call `posthog.reloadFeatureFlags()`, giving time for the person merge to complete.
