PostHog supports OAuth 2.0 for third-party applications to access PostHog on behalf of users. This page explains how to integrate with PostHog's OAuth system.

## Client-Initiated Metadata Discovery (CIMD)

If you're building an application that needs to integrate with PostHog's OAuth, we recommend using **[Client-Initiated Metadata Discovery (CIMD)](https://www.ietf.org/archive/id/draft-parecki-oauth-client-id-metadata-document-07.html)**. This approach allows you to integrate without pre-registering your application with PostHog.

### How CIMD works

With CIMD, your application's **domain acts as your `client_id`**. When a user authorizes your app, PostHog fetches metadata about your client from your domain.

1. **Set up your client metadata**: Host a JSON document at `/.well-known/oauth-client` on your domain (e.g., `https://yourapp.com/.well-known/oauth-client`) containing your client information:

   ```json
   {
     "client_name": "Your App Name",
     "logo_uri": "https://yourapp.com/logo.png",
     "redirect_uris": ["https://yourapp.com/callback"]
   }
   ```

   The `client_name` is required. The `logo_uri` is optional but recommended for a better user experience during authorization.

2. **Discover PostHog's OAuth metadata**: Fetch PostHog's OAuth server configuration from the well-known endpoint:
   - Generic: `https://oauth.posthog.com/.well-known/oauth-authorization-server` (recommended - handles US/EU routing transparently)
   - US Cloud: `https://us.posthog.com/.well-known/oauth-authorization-server`
   - EU Cloud: `https://eu.posthog.com/.well-known/oauth-authorization-server`

   The response contains all the endpoints and capabilities you need, including:
   - `authorization_endpoint` - Where to send users to authorize
   - `token_endpoint` - Where to exchange codes for tokens
   - `scopes_supported` - Available [OAuth scopes](/docs/api#scopes) (same as our REST API)
   - `response_types_supported` - Supported response types

3. **Implement the OAuth flow**: Use your domain as the `client_id` and implement the standard [OAuth 2.0 authorization code flow](https://datatracker.ietf.org/doc/html/rfc6749#section-4.1).

### Example metadata discovery

```bash
curl https://us.posthog.com/.well-known/oauth-authorization-server
```

This returns the OAuth server configuration that your application can use to dynamically configure itself.

## Claiming your OAuth integration

The <SmallTeam slug="growth" /> is working on creating a better, more streamlined process for partners to register and claim OAuth integrations with PostHog. In the meantime, CIMD provides a reliable way to integrate with PostHog's OAuth system.

If you need to register an official OAuth integration or have questions about the integration process, please reach out to the Growth team at [team-growth@posthog.com](mailto:team-growth@posthog.com).

## Token types

PostHog OAuth uses the following token prefixes:

| Token type | Prefix | Description |
|------------|--------|-------------|
| Access token | `pha_` | Short-lived token for API access |
| Refresh token | `phr_` | Long-lived token to obtain new access tokens |

Both token types are automatically revoked if detected by [GitHub secret scanning](/docs/api#github-secret-scanning).
