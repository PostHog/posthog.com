You can track and monitor errors and exceptions in your code by capturing [exception events](/docs/error-tracking/issues-and-exceptions). This can be done automatically when exceptions are thrown in your code, or manually by calling the exception capture method.

## Capturing exceptions

Exceptions are a special type of [event](/docs/data/events) in PostHog. Similar to any other event, they can be captured, customized, filtered, and used in insights for analysis.

To help group exceptions into issues and help you debug them, PostHog automatically captures the following properties:

### Automatic exception capture

If you followed one of our guides to [set up error tracking](/docs/error-tracking/installation) and you [enabled exception auto capture](https://app.posthog.com/project/2/error_tracking?activeTab=configuration), you'll have automatic exception capture enabled.

For browser SDKs (JavaScript Web, React, Next.js, etc.), you can configure which types of errors are automatically captured using the `capture_exceptions` option:

```js
posthog.init('<ph_project_token>', {
    api_host: '<ph_api_client_host>',
    defaults: '<ph_posthog_js_defaults>',
    capture_exceptions: {
      capture_unhandled_errors: true, // default
      capture_unhandled_rejections: true, // default
      capture_console_errors: false // default
  }
})
```

| Option | Default | Description |
|---|---|---|
| `capture_unhandled_errors` | `true` | Captures uncaught errors via `window.onerror`. |
| `capture_unhandled_rejections` | `true` | Captures unhandled promise rejections via `window.onunhandledrejection`. |
| `capture_console_errors` | `false` | Captures calls to `console.error` as exception events. |

You can also pass `capture_exceptions: true` to enable all default options, or `capture_exceptions: false` to disable autocapture entirely.

### Manual exception capture

You can also manually call the exception capture method.

<CalloutBox icon="IconWarning" title="Always use captureException()" type="warning">

Never manually capture `$exception` events using `posthog.capture('$exception', {...})`. Always use `posthog.captureException(error)` instead - it handles the correct event format, stack trace processing, and source map integration automatically.

</CalloutBox>

## Add exception steps

Exception steps let you attach the most relevant actions, checkpoints, and diagnostic state to an exception without emitting extra events. The SDK buffers recent steps in memory and automatically includes them on the next `captureException(...)` call as the `$exception_steps` property on the `$exception` event.

Use this for high-signal breadcrumbs like:

- Checkout or onboarding milestones
- API calls and retries
- Feature flag decisions
- Validation checkpoints
- Job state transitions on the server

### How exception steps work

- Steps are **buffered locally** until an exception is captured.
- Steps are attached to the exception event under **`$exception_steps`**.
- Steps are **not** sent as standalone PostHog events.
- `captureException(...)` includes recent buffered steps by default.
- In PostHog, the steps appear on the related exception in the **session timeline** so you can see what happened immediately before the error.

### Scoping behavior across SDKs

The buffer scope depends on the environment:

- **Frontend SDKs** use a global in-memory recent-step buffer for the current app runtime.
- **Server SDKs** should scope steps to the current request, job, or execution context where possible.
- If a server-side exception is captured **without** an active scope, `captureException(...)` should still send the exception, but buffered steps may be missing.

For server-side SDKs, this means you should set up request or job context before adding steps if you want them to appear on the exception.

### End-to-end example

This example records a few recent checkpoints, then captures the exception. PostHog stores the exception as a `$exception` event with the recent steps attached under `$exception_steps`, and those steps show up in the error's session timeline.

<MultiLanguage>

```javascript file=Web
posthog.addExceptionStep('checkout_started', {
  cart_total: 12900,
  item_count: 3,
})

posthog.addExceptionStep('payment_attempted', {
  provider: 'stripe',
  plan: 'pro',
})

try {
  await confirmCardPayment(clientSecret)
} catch (error) {
  posthog.captureException(error, {
    checkout_id: 'chk_123',
  })
}
```

```javascript file=Node.js
await posthog.withContext({
  distinctId: 'user_123',
  sessionId: '<session_id>',
}, async () => {
  posthog.addExceptionStep('invoice_generation_started', {
    invoice_id: 'inv_123',
    attempt: 1,
  })

  posthog.addExceptionStep('pdf_upload_requested', {
    bucket: 'customer-invoices',
  })

  try {
    await generateInvoicePdf()
  } catch (error) {
    posthog.captureException(error, 'user_123', {
      job_name: 'generate_invoice_pdf',
    })
  }
})
```

```python
import posthog
from posthog import new_context, identify_context, set_context_session

with new_context():
    identify_context('user_123')
    set_context_session('<session_id>')

    posthog.add_exception_step('report_requested', {
        'report_type': 'weekly_summary',
        'workspace_id': 'team_123',
    })

    posthog.add_exception_step('query_started', {
        'model': 'revenue_rollup',
    })

    try:
        generate_report()
    except Exception as e:
        posthog.capture_exception(e, properties={
            'report_id': 'report_123',
        })
```

### Configuring step count and size limits

If your SDK supports exception-step configuration, you should be able to control both:

- the maximum number of buffered steps kept before `captureException(...)` is called
- the maximum serialized size in bytes that can be attached as exception steps on the final exception event

Use a smaller step count when you only care about the most recent checkpoints, or a larger one when longer flows are important for debugging. Use a smaller byte budget when you need tighter control over payload size.

<MultiLanguage>

```javascript file=Web
posthog.init('<ph_project_token>', {
  api_host: '<ph_api_client_host>',
  defaults: '<ph_posthog_js_defaults>',
  capture_exceptions: {
    exception_steps: {
      enabled: true,
      max_queue_size: 20,
      max_bytes: 16384,
    },
  },
})
```

```javascript file=Node.js
const posthog = new PostHog('<ph_project_token>', {
  host: '<ph_api_client_host>',
  captureExceptions: {
    exceptionSteps: {
      enabled: true,
      maxQueueSize: 20,
      maxBytes: 16384,
    },
  },
})
```

```python
posthog.configure(
    '<ph_project_token>',
    host='<ph_api_client_host>',
    capture_exceptions={
        'exception_steps': {
            'enabled': True,
            'max_queue_size': 20,
            'max_bytes': 16384,
        }
    }
)
```

</MultiLanguage>

When the queue is full, SDKs should drop the **oldest** buffered step first so the most recent steps are preserved for the exception.

At capture time, SDKs should then attach the largest in-order set of retained steps that fits within the configured step-count and byte limits.

### How step count and size limits interact

Three limits can control how many steps end up on the final exception event:

1. **Max queue size** limits how many recent buffered steps the SDK keeps in memory.
2. **Max bytes** limits how much serialized exception-step data can be attached to the final `$exception` event.
3. **Max event size** limits how many of those retained steps can actually be attached to the final `$exception` event overall.

At capture time, SDKs should:

- start with the retained buffered steps after queue-size eviction
- keep the steps in chronological order
- add steps to `$exception_steps` in that order
- stop once adding the next step would exceed either the configured attached-step count, the configured byte budget, or the final exception event size

This means the number of attached steps can be smaller than the configured queue size. The included steps should always stay in order, and SDKs should prefer dropping later candidate steps rather than reordering earlier ones.

### Step payload guidelines

Keep step payloads small and focused on the details that help explain the failure.

- Use short step names that describe a checkpoint or action.
- Supported values should be simple JSON types: strings, numbers, booleans, arrays, objects, and `null`.
- Avoid large nested payloads, raw request bodies, or sensitive data.
- SDKs may truncate oversized step payloads before building the final `$exception_steps` array.
- The number of retained candidate steps is controlled by the exception-step max queue size when the SDK exposes that option.
- The number of attached steps may still be lower if adding more steps would exceed the configured attached-step count, byte budget, or final exception event size.
- Steps should be attached in order until one of those limits is reached.

A good rule of thumb is to record the information you'd want to see in the timeline while debugging, but not full logs.

### Automatic exception capture

When an SDK supports exception autocapture, automatically captured exceptions should also include the currently buffered exception steps from the active buffer or scope.

This is especially useful when an error is thrown before your code reaches a manual `catch` block.

### Troubleshooting missing steps

If an exception appears in error tracking but no steps show in the timeline:

- Confirm you're using `addExceptionStep(...)` before the failing code path runs.
- Confirm the exception is sent with `captureException(...)` or SDK autocapture, not `posthog.capture('$exception', ...)`.
- On the server, confirm the exception and steps run inside the same request, job, or execution scope.
- Check that step payloads are not too large and don't contain unsupported values.
- Remember that even valid buffered steps may be omitted if the final exception event reaches its size limit.
- Make sure the exception is linked to a session if you expect to see the steps in a replay timeline.

### Error boundaries

Error boundaries let you catch rendering errors in your component tree and report them to PostHog. We currently support error boundaries for:

- **React** – Use the `PostHogErrorBoundary` component. See the [React error boundaries guide](/docs/error-tracking/installation/react#set-up-error-boundaries).
- **React Native** – Use the `PostHogErrorBoundary` component. See the [React Native error boundaries guide](/docs/error-tracking/installation/react-native#set-up-error-boundaries).

## Customizing exception capture

Like all data, the better **quality** of your exception events, the more **context** you'll have for debugging and analysis. Customizing exception capture helps you do this. 

Customizing exception capture lets you override exception properties to influence how they're grouped into issues. 

Equally important, you can customize properties on the exceptions to help you configure rules for [automatic issue assignment](/docs/error-tracking/assigning-issues#automatic-issue-assignment), [alerts](/docs/error-tracking/alerts), [issue grouping](/docs/error-tracking/grouping-issues). They can also be used in analysis in [insights](/docs/product-analytics/insights), [dashboards](/docs/product-analytics/dashboards), and [data warehouse queries](/docs/data-warehouse/start-here).

### Customizing exception properties 

#### During manual capture

When capturing exceptions manually, passing properties to the capture exception method adds them to the event just like any other PostHog event.

You can also override the [fingerprint](/docs/error-tracking/fingerprints) to [group](/docs/error-tracking/issues-and-exceptions#how-are-issues-grouped) exceptions together at capture time.

Here are some examples of how to override exception properties:

<MultiLanguage>

```javascript file=Web
try {
  // ...
} catch (error) {
 posthog.captureException(error, { 
      "custom_property": "custom_value",
      // Optional: you can also override generated properties like fingerprint
      "$exception_fingerprint": "CustomExceptionGroup",
    });
}
```

```javascript file=Node.js
try {
  // ...
} catch (error) {
 posthog.captureException(error, 'user_123', { 
      "custom_property": "custom_value",
      // Optional: you can also override generated properties like fingerprint
      "$exception_fingerprint": "CustomExceptionGroup",
    });
}
```

```python
import posthog

posthog.api_key = '<ph_project_token>'
posthog.host = '<ph_client_api_host>'

# With context
with posthog.new_context():
    posthog.identify_context(distinct_id="user_123")

    # When using context, you define properties with tags
    posthog.tag("custom_property", "custom_value")

    # Optional: you can also override generated properties like fingerprint
    posthog.tag("$exception_fingerprint", ["custom_fingerprint"])

    posthog.capture_exception(Exception("Test custom exception"))

# Or without context
posthog.capture_exception(
    Exception("Test custom exception without context"),
    distinct_id="user_123",
    properties={
        "$exception_type": "Custom exception without context",
        "$exception_fingerprint": ["custom_fingerprint without context"],
        "custom_property": "custom_value",
    },
)
```

```ruby
begin
  subscriptions.activate(user_id, subscription_id)
rescue => e
  posthog.capture_exception(
    e,
    distinct_id: 'user_123',
    properties: {
      subscription_id: subscription_id,
      # You can also *optionally* override generated properties like fingerprint
      '$exception_fingerprint': 'CustomExceptionGroup'
    }
  )
end
```

```go
fingerprint := "CustomExceptionGroup"

exception := posthog.Exception{
    DistinctId: "user_123",
    Timestamp:  time.Now(),
    ExceptionList: []posthog.ExceptionItem{
        {
            Type:  "ActivationError",
            Value: err.Error(),
        },
    },
    // Optional: override the fingerprint for custom grouping
    ExceptionFingerprint: &fingerprint,
}
client.Enqueue(exception)
```

</MultiLanguage>

  In [JavaScript Web](/docs/libraries/js) and [Node.js](/docs/libraries/node) SDKs, you can override the default properties by passing a `before_send` callback. This callback is called before any exception is captured.

  ```javascript 
  posthog.init('<ph_project_token>', {
  api_host:'<ph_client_api_host>',
  defaults: '<ph_posthog_js_defaults>',
  before_send: (event) => {
    if (event && event.event === "$exception") {
      const exceptionList = event.properties?.["$exception_list"] || []
      const exception = exceptionList.length > 0 ? exceptionList[0] : null;

      if (exception) {
        event.properties["custom_property"] = "custom_value"
        // Optional: you can also override generated properties like fingerprint
        event.properties["$exception_fingerprint"] = "MyCustomGroup"
        // ... and any other properties you want to override
      }    
    }
    return event
    }
  })
  ```

### Capturing properties for custom issue grouping rules

Other than grouping with [custom fingerprints](#customizing-exception-properties), you can also set custom properties on the exception to help you group exceptions together using [issue grouping rules](/docs/error-tracking/grouping-issues). 

For example:

- Setting fields like `db_transaction` to group exceptions together for a specific database transactions.
- Setting `feature_name`, `service_name`, or `service_version` to group exceptions together for a specific features and services.
- Setting `intent` to group exceptions due to common interactions like accessing storage or network.

It's important to think about your grouping rules when you configure exception capture. You cannot group exceptions together if you don't set some common properties between them. 

Note that grouping issues means that all exceptions will be grouped together under a **single issue**. You **cannot** group exceptions into multiple issues, but you can still [filter on custom properties](/docs/error-tracking/monitoring#finding-specific-issues) across multiple issues without grouping them.

## Common problems

### Exception steps don't appear in the session timeline

If you can see the exception in error tracking but not its steps in the timeline, the most common causes are:

- The steps were recorded after the exception was captured.
- The server-side code didn't have an active request or job scope.
- The exception wasn't linked to the same user or session as the replay.
- The step payload was too large and got truncated or dropped.

Start by verifying identification and session linkage, then confirm your steps are being added immediately before the risky operation.


### Don't use `posthog.capture()` for exceptions

<CalloutBox icon="IconWarning" title="Don't do this" type="caution">

  **Do not** use `posthog.capture('$exception', { ... })` and try to attach exception metadata yourself (e.g. `$exception_stack_trace_raw`, `$exception_type`, `$exception_message`). This approach fails in the vast majority of cases because the exception event schema is strict and the SDK does not normalize or validate these properties when you use `capture()`.

  **Do this instead:** use `posthog.captureException(error)`. It accepts an `Error` object (or your runtime's equivalent), extracts stack trace, type, and message correctly, and sends a valid `$exception` event with all required metadata. You can still pass optional properties as the second argument.

</CalloutBox>
