Drop rules filter out logs at ingestion before they're stored. This helps you control costs and reduce noise without changing your application code.

Go to [**Logs** > **Settings**](https://app.posthog.com/logs/settings) to create and manage drop rules.

PostHog supports three rule types:

- **Drop** - Drop logs that match specific filters
- **Drop by severity** - Sample or drop entire severity levels (debug, info, warn, error)
- **Rate limit** - Cap log volume (in KB/s) for logs matching specific filters

Rules are evaluated in priority order. The first matching rule wins.

## Drop

Drop all logs that match specific filters, like health checks or noisy services.

1. Go to [**Logs** > **Settings**](https://app.posthog.com/logs/settings) and click **New rule**
2. Select **Drop** as the action type
3. Add filters to match the logs you want to drop

Use the filter bar to build conditions based on any log attribute. Common filters include:

| Filter | Example | Effect |
|--------|---------|--------|
| `service.name` | `equals "health-checker"` | Drop logs from a specific service |
| `http.route` | `contains "/internal/"` | Drop internal API routes |
| `container.name` | `is set` | Drop logs from containerized workloads |

The **volume preview** shows how many logs would match your filters over the last 24 hours, broken down by service. Use this to verify your rule targets the right logs before saving.

## Drop by severity

Sample or drop logs based on severity level. You can set each level independently to keep all, drop all, or sample a percentage.

1. Go to [**Logs** > **Settings**](https://app.posthog.com/logs/settings) and click **New rule**
2. Select **Drop by severity** as the rule type
3. Configure an action for each severity level:
   - **Keep** - Store all logs at this level
   - **Drop** - Discard all logs at this level
   - **Sample** - Keep a percentage (e.g., 10% keeps 1 in 10 logs)

**Example configuration:**

| Level | Action | Effect |
|-------|--------|--------|
| Debug | Drop | Drop all debug logs |
| Info | Sample 10% | Keep 10% of info logs |
| Warn | Keep | Keep all warnings |
| Error | Keep | Keep all errors |

You can also configure **Always keep** conditions to preserve important logs regardless of severity:

- **Status code ≥** - Keep logs with HTTP status codes at or above a threshold
- **Latency >** - Keep logs where request latency exceeds a threshold in milliseconds

## Rate limit

Cap log volume (in KB/s) for logs matching specific filters. Logs above the limit are dropped at ingestion.

1. Go to [**Logs** > **Settings**](https://app.posthog.com/logs/settings) and click **New rule**
2. Select **Rate limit** as the action type
3. Set the rate limit in KB/s, MB/s, or GB/s (minimum 1 KB/s, maximum 1 GB/s)
4. Add filters to specify which logs the rate limit applies to

**How rate limiting works:**

PostHog uses a token bucket algorithm based on log volume (bytes), not count:

- The bucket fills at your configured rate (e.g., 5 MB/s)
- The bucket holds up to 10× your rate limit as burst capacity
- Each log consumes tokens based on its size in bytes
- When the bucket is empty, excess logs are dropped

This allows short traffic spikes while keeping your average volume at or below the configured rate.

The **volume preview** shows matching log volume over the last 24 hours. A dashed line indicates your rate limit threshold – logs above this line would be dropped. If the threshold line is above the chart, no logs would be dropped under current traffic patterns.

**Example:**

| Setting | Value | Effect |
|---------|-------|--------|
| Rate limit | 5 MB/s | Store up to 5 MB/second of matching logs |
| Filter | `service.name equals "payment-api"` | Only rate limit this service |

With these settings, if `payment-api` emits 10 MB of logs in one second, approximately 5 MB are stored and the rest are dropped.

**Configuration limits:**

- Rate limit: 1 KB/s to 1 GB/s
- Burst capacity: automatically set to 10× your rate limit

## Volume preview

When creating or editing a drop or rate limit rule, a volume preview shows how your filters would affect logs from the last 24 hours.

The preview displays:

- **Sparkline chart** – matching log volume over time, broken down by service (top 10 services shown)
- **Total count or volume** – matching logs in the preview period
- **Dashed threshold line** – for rate limit rules, shows where logs would be dropped

Use this to verify your filters target the right logs before saving.

## Rule priority

Rules are evaluated in the order shown in the settings UI. Drag rules to reorder them. The first matching rule determines whether a log is kept or dropped.

If no rules match, logs are kept.

## Monitoring rule impact

The rules table shows an **Impact (24h)** column with how many logs each rule dropped in the last 24 hours. Use this to see which rules are actively filtering logs and adjust as needed.
