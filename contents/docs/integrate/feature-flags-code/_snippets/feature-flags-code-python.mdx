There are 2 steps to implement feature flags in Python:

### Step 1: Evaluate the feature flag value

Call `evaluate_flags()` to fetch all flags for a user in a single request. It returns a snapshot you can use for branching and pass to `capture()`, ensuring events record the exact flag values your code acted on.

```python
flags = posthog.evaluate_flags(
    'distinct_id_of_your_user',
    person_properties={'plan': 'enterprise'}
)

# Boolean check
if flags.is_enabled('flag-key'):
    # Do something differently for this user

# Get variant value for multivariate flags
variant = flags.get_flag('flag-key')
if variant == 'variant-key':
    # Handle specific variant

# Get payload
payload = flags.get_flag_payload('flag-key')
```

#### Additional parameters

| Parameter | Description |
|-----------|-------------|
| `groups` | Group associations for group-scoped flags (e.g., `{'company': 'acme-inc'}`) |
| `person_properties` | Person properties to use for evaluation |
| `group_properties` | Group properties keyed by group type |
| `only_evaluate_locally` | If `True`, only evaluate flags locally without server fallback. Flags that can't be evaluated locally are omitted from the result. |
| `disable_geoip` | Whether to disable GeoIP lookup |
| `flag_keys` | List of flag keys to fetch. When set, only these flags are requested from the server. |
| `device_id` | Device ID for experience-continuity flags to match users across distinct ID changes |

#### The `FeatureFlagEvaluations` snapshot

The snapshot returned by `evaluate_flags()` provides these methods:

| Method | Description |
|--------|-------------|
| `is_enabled(key)` | Returns `True` if the flag is enabled, `False` otherwise |
| `get_flag(key)` | Returns the variant string for multivariate flags, `True` for enabled boolean flags, `False` for disabled flags, or `None` if the flag wasn't returned |
| `get_flag_payload(key)` | Returns the payload associated with a flag |

#### Scoping which flags to fetch

Use the `flag_keys` parameter to limit which flags are requested from the server:

```python
flags = posthog.evaluate_flags(
    'distinct_id_of_your_user',
    flag_keys=['new-dashboard', 'checkout-flow']
)
```

**Boolean feature flags:**

```python
is_my_flag_enabled = posthog.feature_enabled('flag-key', 'distinct_id_of_your_user')

    # Optional: fetch the payload
```

**Multivariate feature flags:**

```python
enabled_variant = posthog.get_feature_flag('flag-key', 'distinct_id_of_your_user')

    # Optional: fetch the payload

```

To include feature flags with events, pass the `flags` snapshot from `evaluate_flags()` to `capture()`:

```python
flags = posthog.evaluate_flags('distinct_id_of_your_user')

if flags.is_enabled('new-dashboard'):
    render_new_dashboard()

# Pass the flags snapshot to capture - no additional network call
posthog.capture(
    'page_viewed',
    distinct_id='distinct_id_of_your_user',
    flags=flags
)
```

This ensures the event carries the exact flag values your code branched on.

#### Filtering which flags to attach

Use filter helpers to control which flags are attached to an event:

```python
# Only attach flags your code actually checked
posthog.capture('page_viewed', distinct_id='user_123', flags=flags.only_accessed())

# Only attach specific flags
posthog.capture('checkout_started', distinct_id='user_123', flags=flags.only(['checkout-flow']))
```

> **Note:** `only_accessed()` returns an empty snapshot if no flags were checked via `is_enabled()` or `get_flag()`. Make sure to access the flags you need before calling `only_accessed()`.

#### Alternative methods

**Method 1: Include the `$feature/feature_flag_name` property**

In the event properties, include `$feature/feature_flag_name: variant_key`:

```python
posthog.capture(
    "event_name",
    distinct_id="distinct_id_of_the_user",
    properties={
        "$feature/feature-flag-key": "variant-key"  # replace feature-flag-key with your flag key. Replace 'variant-key' with the key of your variant
    },
)
```

### Fetching all flags for a user

Use `evaluate_flags()` to fetch all flags for a user in a single request. See [Step 1](#step-1-evaluate-the-feature-flag-value) above for details.

The following methods also return all flags but as raw dictionaries:

```python
posthog.get_all_flags('distinct_id_of_your_user')
posthog.get_all_flags_and_payloads('distinct_id_of_your_user')
```

### Sending `$feature_flag_called` events

Capturing `$feature_flag_called` events enables PostHog to know when a flag was accessed and provide [analytics and insights](/docs/product-analytics/insights) on the flag. By default, events are sent when:

1. You call `flags.is_enabled()` or `flags.get_flag()` on a `FeatureFlagEvaluations` snapshot, OR
2. You call `posthog.get_feature_flag()` or `posthog.feature_enabled()`, AND
3. It's a new user, or the value of the flag has changed.

The SDK deduplicates events automatically, so calling `is_enabled()` multiple times for the same flag only sends one event.

> *Note:* Deduplication happens in a local cache. If you reinitialize the PostHog client, the cache resets, causing `$feature_flag_called` events to be sent again. PostHog handles duplicates, so this won't affect your analytics.

You can disable automatically capturing `$feature_flag_called` events. For example, when you don't need the analytics, or it's being called at such a high volume that sending events slows things down.

To disable it, set the `send_feature_flag_events` argument in your function call, like so:

```python
is_my_flag_enabled = posthog.feature_enabled('flag-key', 'distinct_id_of_your_user', send_feature_flag_events=False)
# will not send `$feature_flag_called` events
```

### Request timeout

You can configure the `feature_flags_request_timeout_seconds` parameter when initializing your PostHog client to set a flag request timeout. This helps prevent your code from being blocked in the case when PostHog's servers are too slow to respond. By default, this is set at 3 seconds.

```python
posthog = Posthog('<ph_project_token>',
    host='<ph_client_api_host>'
    feature_flags_request_timeout_seconds=3 // Time in second. Default is 3
)
```

### Error handling

When using the PostHog SDK, it's important to handle potential errors that may occur during feature flag operations. Here's an example of how to wrap PostHog SDK methods in an error handler:

```python
def handle_feature_flags(distinct_id):
    try:
        flags = posthog.evaluate_flags(distinct_id)
        return flags
    except Exception as e:
        print(f"Error fetching feature flags: {str(e)}")
        raise e

# Usage example
try:
    flags = handle_feature_flags('user-123')
    if flags.is_enabled('new-feature'):
        # Implement new feature logic
    else:
        # Implement old feature logic
except Exception as e:
    # Handle the error at a higher level
```
