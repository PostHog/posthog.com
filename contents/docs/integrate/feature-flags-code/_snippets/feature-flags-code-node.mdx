There are two steps to implement feature flags in Node:

### Step 1: Evaluate flags once

Call `client.evaluateFlags()` once for the user, then read values from the returned snapshot.

For server-side applications, use `evaluateFlags()` to evaluate all feature flags for a user in a single call. This is the recommended approach because it:

- Makes a single `/flags` request instead of one per flag check
- Ensures events capture the exact flag values your code branched on
- Lets you filter which flags get attached to events

```node
const flags = await client.evaluateFlags('distinct_id_of_your_user', {
    groups: { company: 'id:5' },
    personProperties: { plan: 'enterprise' },
    groupProperties: { company: { name: 'Acme' } }
})

if (flags.isEnabled('new-dashboard')) {
    renderNewDashboard()
}

// Attach the flag values to the captured event
client.capture({
    distinctId: 'distinct_id_of_your_user',
    event: 'page_viewed',
    flags
})
```

#### Getting flag values and payloads

Use `isEnabled()` to check if a flag is enabled, `getFlag()` to get the variant value, and `getFlagPayload()` to get the payload:

```node
const flags = await client.evaluateFlags('distinct_id_of_your_user')

// Boolean check
if (flags.isEnabled('new-feature')) {
    // Feature is enabled
}

// Get the variant value (returns true, false, or variant string)
const variant = flags.getFlag('experiment-flag')
if (variant === 'control') {
    // Show control experience
} else if (variant === 'test') {
    // Show test experience
}

// Get the payload associated with a flag
const payload = flags.getFlagPayload('feature-with-payload')
```

#### Filtering flags attached to events

You can filter which flags get attached to captured events using `onlyAccessed()` or `only()`:

```node
const flags = await client.evaluateFlags('distinct_id_of_your_user')

if (flags.isEnabled('checkout-v2')) {
    processCheckoutV2()
}

// Only attach flags that were actually checked via isEnabled() or getFlag()
client.capture({
    distinctId: 'distinct_id_of_your_user',
    event: 'checkout_completed',
    flags: flags.onlyAccessed()
})

// Or specify exactly which flags to attach
client.capture({
    distinctId: 'distinct_id_of_your_user',
    event: 'checkout_completed',
    flags: flags.only(['checkout-v2', 'payment-method'])
})
```

#### Scoping the flags request

To reduce the response payload, you can scope the `/flags` request to specific flag keys:

```node
const flags = await client.evaluateFlags('distinct_id_of_your_user', {
    flagKeys: ['checkout-v2', 'payment-method']
})
```

#### Exception capture with flag context

When capturing exceptions, you can pass the flags snapshot to include feature flag context with your error tracking events:

```node
const flags = await client.evaluateFlags('distinct_id_of_your_user')

try {
    if (flags.isEnabled('new-checkout')) {
        processNewCheckout()
    }
} catch (e) {
    // Exception event includes the flag context
    client.captureException(e, 'distinct_id_of_your_user', undefined, undefined, flags)
}
```

### Request timeout

You can configure the `feature_flag_request_timeout_ms` parameter when initializing your PostHog client to set a flag request timeout. This helps prevent your code from being blocked if PostHog's servers are too slow to respond. By default, this is set to 3 seconds.

```js
const client = new PostHog('<ph_project_token>', {
    api_host: '<ph_client_api_host>',
    feature_flag_request_timeout_ms: 3000, // Time in milliseconds. Defaults to 3000 (3 seconds).
})
```
