---
title: Caching
---

Caching stores query results temporarily so repeated calls don't re-run the query. This is different from [materialization](/docs/endpoints/materialization), which pre-computes and stores results on a schedule.

## How caching works

When you execute an endpoint:

1. PostHog checks if there's a cached result that's still within the [data freshness](#data-freshness) window
2. If yes, it returns the cached result immediately
3. If no, it runs the query and caches the result

## Data freshness

Data freshness controls how fresh the data returned by your endpoint should be. You configure it in the **Configuration** tab by choosing from one of the predefined time intervals:

| Interval        | Seconds |
| --------------- | ------- |
| 15 minutes      | 900     |
| 30 minutes      | 1,800   |
| 1 hour          | 3,600   |
| 6 hours         | 21,600  |
| 12 hours        | 43,200  |
| 1 day (default) | 86,400  |
| 7 days          | 604,800 |

How data freshness applies depends on whether your endpoint is materialized:

- **Non-materialized endpoints** – Data freshness sets the cache TTL. PostHog caches query results for the duration you choose. Once the window expires, the next request re-runs the query and caches the new result.

- **Materialized endpoints** – Data freshness controls both the materialization sync schedule and the cache TTL. PostHog syncs (re-materializes) your query results on a schedule matching the interval you select. Between syncs, the cached materialized result is served. The cache TTL is dynamically set to the remaining time until the next sync is due.

See [endpoint execution](/docs/endpoints/execution) for more detail on how this can affect your results.

## Caching vs materialization

| Feature  | Caching                           | Materialization                 |
| -------- | --------------------------------- | ------------------------------- |
| Storage  | Temporary                         | Persistent (S3)                 |
| Refresh  | On demand                         | Scheduled                       |
| Best for | Moderate traffic, real-time needs | High traffic, expensive queries |

Use caching when you need relatively fresh data. Use materialization when query speed matters more than freshness.
