Today, [PostHog Code](/code) enters beta. It's a desktop app that runs coding agents on top of your product data. The obvious stuff ships itself. The tricky stuff shows up as a prioritized to-do list for you to steer.

We built it to chase one idea: the self-driving product. A product that spots its own problems, fixes them, checks the fix worked, and remembers what it learned. We're not there yet. But we're closer than you might expect.

## Six levels of autonomy, but for products

The Society of Automotive Engineers has a framework for self-driving cars: six levels, from "you do everything" to "you can take a nap." It maps well onto building products with AI.

- **Level 0 – No automation.** Dashboards, SQL, docs. Engineers do everything. This was most of what shipped before 2024.
- **Level 1 – Engineer assistance.** The agent answers narrow questions and writes short snippets when you point at the file. Ambiguity breaks it.
- **Level 2 – Partial automation.** The agent handles vague questions, stitches queries together, and writes whole functions. You still decide what to ask.
- **Level 3 – Conditional automation.** The agent summarizes thousands of signals and drafts a PR. You review instead of originate.
- **Level 4 – High automation.** The agent understands your code, your data, and the world around it. It proposes and ships work on its own, inside guardrails you set.
- **Level 5 – Full automation.** The loop runs itself. Signals come in, work ships, evals check it, memory updates. You set strategy.

[PostHog AI](/ai) crossed into Level 3 in late 2025. The [PostHog MCP](/docs/model-context-protocol) makes Level 4 possible today. Level 5 is what we're building toward.

## Why vibe-coding tools plateau at Level 3

A self-driving car isn't a vehicle that goes where you point it (that's cruise control). A self-driving product isn't just a really good coding agent. To reach Level 5, it has to sense what's happening, decide what to do, do it, check if it worked, and adjust next time — all without being asked. 

The *self* in self-driving isn't autonomy from the engineer. It's autonomy from instruction.

Vibe-coding tools aren't aiming for that. They're aiming for speed per instruction – and they've gotten *very* good at it. Point Lovable, Replit or [v0](/blog/posthog-v0-integration) at a goal and they'll ship a working app before lunch. That's a legitimate Level 3 experience for *building* software, and it's why [AI has quietly eaten the no-code category](/blog/ai-is-killing-no-code-experiments).

![On Wednesdays we do trace reviews](https://res.cloudinary.com/dmukukwp6/image/upload/aps0un_8b5268c19e.jpg)

The problem is what happens after lunch. In a typical week, PostHog customers generate 100,000+ failed queries, 2.6 million new error tracking issues, and 47 million raw exceptions. Each one is a signal an agent could act on. None of it is visible to a tool whose worldview ends at the last git push.

To move from cruise control to self-driving, an agent needs five things: **tools, skills, signals, memory,** and **evaluation**. Our [AI engineering handbook](/handbook/engineering/ai/ai-platform) goes deep on each. Here's the short version.

### 1. Tools – what the agent can do

Tools are small, specific actions the agent can take. At PostHog we treat them as [atomic capabilities](/handbook/engineering/ai/implementation#mcp-tools-vs-skills) — things like `create_insight` or `read_taxonomy`. The latter does a lot of heavy lifting. It lets the agent check what events and properties actually exist before it writes a query or an instrumentation PR.

### 2. Skills – how to get the job done

If tools are the fork and knife, skills are the recipe. A [skill](/handbook/engineering/ai/writing-skills) ties tools, docs, and rules into a playbook. You can see the gap in our own data: `docs-search` is the most-called tool on PostHog's MCP server, at ~28K calls a month.

PostHog Code has skills for the workflows we see most – instrumenting events, auditing flags, adding error tracking. Writing a skill feels like writing a doc, and [most engineers would rather ship the feature](/newsletter/agent-first-product-engineering) than document it. But for an agent, the skill *is* the feature.

### 3. Signals – when the work should happen

Tools and skills cover the *what*. Signals cover the *when*. PostHog Code sits on top of your product data, so the pattern itself is the prompt. Raw observations get grouped, enriched, and turned into concrete plans. You open a pre-framed to-do list instead of triaging a noisy inbox.

### 4. Memory – what the agent already knows

Signals say something's happening now. Memory says what happened last time. Without it, the agent [re-runs work it's already done](/blog/optimizing-agent-cost) and re-opens the same PR every Tuesday.

### 5. Evaluation – did it actually work?

The loop doesn't close without this. [Testing AI agents](/blog/testing-ai-agents) looks nothing like testing normal software, and "it runs without erroring" is not a pass. PostHog Code schedules evals as long-running Temporal jobs, so the check runs hours or days after the PR merges. The same dashboard, funnel, experiment, or [LLM-as-a-judge eval](/blog/stop-ai-slop) that fired the signal gets re-queried. If the metric didn't move — or moved the wrong way – the agent reverts or reopens the work.

## The product autonomy loop

Stitch the five together and you get a loop:

*Collect data → cluster signals → check memory → notify workers → do work → evaluate → write back to memory.*

That's what a self-driving product runs on. You can't do this reliably inside a general-purpose coding agent, because the critical signals live somewhere else. For a lot of companies, that somewhere else is PostHog.

## What PostHog Code does for you

We've spent years building the stack this loop needs:

- **Data** – [errors](/docs/error-tracking/start-here), [session replays](/docs/session-replay/start-here), [LLM traces](/docs/llm-analytics/start-here), [experiments](/docs/experiments/start-here), [feature flags](/docs/feature-flags/start-here), revenue. What real users are actually doing.
- **Warehouse** – product events plus anything you sync from Stripe, Salesforce, Snowflake, BigQuery, or flat files. ClickHouse today, DuckDB soon.
- **Orchestration** – Temporal for long-running jobs that outlive any one agent session.
- **Sandboxes** – cloud agents that run code safely outside of production.
- **Devtools** – the things product engineers actually reach for.

PostHog Code is the interface that ties it all together. It does two things.

**It ships PRs for the obvious stuff.** Something breaks, the agent pulls in context, writes the fix, opens a PR, and schedules an evaluation. If the metric moves the wrong way, it reverts itself.

**It frames the judgement calls for you.** Ten support tickets about a confusing checkout? PostHog Code checks the replays, quantifies the drop-off, and drafts a PR. You iterate, kill, or deploy.

Under the hood, it's a multi-harness, multi-agent orchestrator. You can run Claude Code and Codex side by side, each on its own task, with their own model and settings. A split-screen Command Center lets you run up to nine agents in parallel – our engineers call it *dopamine mode* (you'll understand why).

Git and GitHub are native. Branches, commits, PRs, and review all happen inside the app, with AI-written branch names, commit messages, and PR descriptions. Long jobs run in the cloud so your laptop stays free; local worktrees handle the rest.

The PostHog side has built-in skills for instrumenting events, flags, and experiments, plus the PostHog MCP for measuring impact, debugging errors, and building dashboards. Plug in your own MCP servers (Attio, Canva, Atlassian, Notion, Monday) and pull context from Linear, Zendesk, and GitHub into the same pipeline as your product data.

You get a prioritized to-do list instead of a noisy one, and automatic fixes for the stuff that doesn't need your attention.

Product engineers [keep telling us this is the missing piece](/newsletter/agent-first-product-engineering) – an agent that understands your codebase *and* your product. We're not at Level 5 yet. But the gap closes every week, and you get to watch it happen from the driver's seat.
